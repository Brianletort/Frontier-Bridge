# Running frontier MoE models on 16 GB-VRAM / 128 GB-DDR laptops

Generated by `frontier runbook render` from `runbooks/gpu-laptop-128gb.yaml` — do not edit by hand.

**Hardware class:** gpu_laptop_128gb — Linux laptops and small-form workstations with a discrete NVIDIA GPU in the 12-24 GB VRAM class and 96 GB or more of system RAM (RTX 4090/5000 Ada mobile class). The hierarchy has three levels: VRAM holds attention, KV, and hot experts; system RAM holds the routed expert pool (--n-cpu-moe); NVMe is the streaming backstop. Big-DDR laptops are a sleeper class — the expert pool lives in RAM, not VRAM, so 16 GB of VRAM is not the ceiling it appears to be.


Committed profiles known to match: `rtx4090_laptop_16gb_128ram`

## Does this runbook apply to your machine?

Run `frontier runbook match` — it profiles your machine and checks it against every runbook's requirements. This class requires:

- memory, class in device_local, capacity_gb {'min': 12, 'max': 24}
- memory, class in system, capacity_gb {'min': 96}
- compute, class in gpu

## Bring-up

1. Bootstrap and check readiness (doctor lists every gap with its fix).

   ```bash
   git clone https://github.com/Brianletort/Frontier-Bridge.git
cd Frontier-Bridge && ./scripts/bootstrap.sh

   ```

   *Check:* doctor should show nvidia-smi present. If llama-server is missing, the next step builds it.


2. Build CUDA llama.cpp with a recorded commit (the runtime pin verified rows require).

   ```bash
   ./scripts/build_llama_cpp_cuda.sh
   ```

   *Check:* Note the printed runtime commit — every bench run takes it via --runtime-commit. Details in docs/linux_runtime.md.


3. Profile the machine from the filesystem the models will live on.

   ```bash
   frontier detect -o hardware_profiles/local/my_laptop.yaml
   ```

   *Check:* Expect a discrete topology: a device_local vram node attached to the GPU, a system memory node, and an nvme node. Laptop SSDs throttle under sustained load — re-run the disk bench after a long read to catch it.


4. Measure the SSD at expert-slice granularity (the streaming floor for models larger than RAM).

   ```bash
   frontier bench ssd-stream --output results/local/ssd_expert_read.jsonl
   ```

5. Find the offload stability boundary before benching (which experts fit where is this class's central tuning knob).

   ```bash
   frontier bench sweep-offload --model ~/models/<gguf> --values 8,16,24,32 --output results/local/offload_sweep.jsonl
   ```

## Troubleshooting

- **llama-server built but CUDA is not used at runtime.** — check: The build found nvcc but the driver/toolkit versions may mismatch — check the llama-server startup log for the CUDA device line.

  Fix: Rebuild after aligning driver and toolkit; scripts/build_llama_cpp_cuda.sh prints both.
- **Decode stalls or the machine swaps under load.** — check: The --n-cpu-moe value is past the RAM working set, or another process holds memory. Re-run the sweep-offload experiment and pick the last stable point.

  Fix: Lower the offload value; leave ~10% of RAM free for the OS.
- **SSD numbers look great cold and terrible after ten minutes.** — check: Laptop NVMe thermal throttling — common in this class.
  Fix: Re-run frontier bench ssd-stream after sustained load and record the worst figure as expert_read_gbps; the planner should see the throttled number, not the brochure number.
